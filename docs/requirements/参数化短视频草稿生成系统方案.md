# 参数化短视频草稿生成系统方案

> 版本：v0.1  
> 状态：头脑风暴汇总 / 产品与技术方案草案  
> 目标：将可标准化的爆款视频模板沉淀为高代码编排资产，最终生成剪映草稿 DSL/JSON，并通过本地工具转换为剪映可打开的草稿文件。  
> 适用对象：产品、研发、AI Coding、模板设计、内容运营、测试验收。

---

## 1. 背景与目标

我们当前的目标不是做一个类似剪映的完整视频剪辑软件，也不是做一个完全依赖 AI 文生视频的生成平台，而是构建一套面向批量内容生产的 **参数化短视频草稿生成系统**。

该系统的核心思路是：

> 把可标准化、可复用、可批量生产的爆款视频结构，拆解为固定框架和动态参数。用户只需要补齐部分参数，系统即可自动生成风格一致、节奏稳定、素材可控的剪映草稿。

最终用户拿到的不是直接渲染完成的视频，而是一个可以在剪映中继续预览、二次编辑、导出的视频草稿。

---

## 2. 产品定位

### 2.1 不做什么

本产品不以替代剪映为目标，因此不优先做以下能力：

- 不做完整在线时间线编辑器
- 不做类似 PR / Final Cut 的自由剪辑工具
- 不做 Vlog / 真人口播 / 随拍混剪这类高度不可控内容
- 不做复杂多轨自由拖拽
- 不把 AI 文生视频作为第一阶段核心能力
- 不让用户无限制调整素材、节奏、转场、轨道

这些能力虽然看起来强大，但会把产品拖入“通用剪辑器”的复杂度中，不适合当前阶段。

### 2.2 要做什么

本产品要做的是：

> 面向批量内容生产的短视频模板工作流系统。

更具体地说：

> 将爆款短视频拆解成可复用的模板 Spec，通过高代码编排和通用能力插件，自动生成剪映草稿 DSL/JSON，再由本地工具转换为剪映可打开的本地草稿文件。

### 2.3 一句话定位

可以用下面几句话描述产品：

- 把爆款视频变成可复用工作流。
- 不是剪视频，而是批量生产标准化短视频草稿。
- 填写动态参数，即可批量生成同风格剪映草稿。
- 面向剪映生态的参数化短视频草稿生成引擎。

## 3. 核心业务价值

### 3.1 对用户的价值

用户不需要从零剪辑，只需要：

1. 选择一个爆款视频模板
2. 填写动态参数
3. 上传或选择必要素材
4. 点击生成
5. 下载草稿包
6. 在剪映中打开预览、微调、导出

用户获得的是：

- 更低的视频制作门槛
- 更快的内容产出速度
- 更稳定的视觉风格
- 更适合矩阵号 / 批量测试的内容变体
- 可在剪映中二次编辑的草稿文件

### 3.2 对团队的价值

对产品和研发团队而言，该系统的核心资产不是某一个低代码节点流，而是：

- 爆款模板 Spec
- 参数 Schema
- 高代码 Builder
- 通用能力插件
- 剪映草稿 Adapter
- 模板测试和验收规范

这些资产可以被 Git 管理、被测试、被复用，也更适合后续 AI Coding 参与开发维护。

## 4. 为什么不继续使用低代码工作流

此前我们已经在类似扣子工作流、n8n 工作流中进行了实践沉淀，证明这条链路是可行的。但随着模板变复杂，低代码工作流开始暴露出明显问题。

### 4.1 节点多，维护成本高

一个完整的视频生成流程会涉及：

- 参数输入
- 文案生成
- 文案仿写
- 文件转文本
- 分镜生成
- 字幕拆分
- 中英文翻译
- 配图生成 / 检索
- 图片去重
- 语音合成
- 音频时间线处理
- 图片轨道处理
- 字幕轨道处理
- 关键帧动画
- 剪映草稿生成
- 草稿包导出

在低代码系统中，这些能力会被拆成大量节点。当节点数量增加后，流程图会变成“节点迷宫”，排查、复用、重构都很困难。

### 4.2 平台升级导致节点不可用

低代码平台依赖大量平台内置节点和第三方插件。一旦平台升级、插件升级、字段变更、鉴权变化，就可能导致原本可用的工作流不可用。

对标准化视频生产系统来说，这种不稳定性不可接受。

### 4.3 工程化测试困难

视频模板上线前需要验证：

- 输入参数是否合法
- 字幕是否越界
- 素材是否缺失
- 时间线是否错位
- 音频是否对齐
- 草稿能否被剪映打开
- 剪映能否正常预览和导出
- 不同输入样例是否稳定生成

这些测试在高代码工程中可以写成单元测试和集成测试，但在低代码节点系统中很难系统化执行。

### 4.4 AI Coding 不友好

未来模板迭代会大量依赖 AI Coding。如果流程藏在低代码节点图中，AI 很难理解、修改、测试。

如果每个模板都有：

- `spec.md`
- `input_schema.json`
- `builder.py`
- `sample_input.json`
- `handoff.md`
- `tests/`

AI Coding 就可以更稳定地参与模板开发、重构、迁移和测试。

因此，低代码适合早期探索，高代码适合长期产品化。

## 5. 从历史扣子工作流中抽象出的能力

参考过往扣子工作流源码，当前视频生产链路已经不仅仅是“生成剪映草稿”，而是包含了多类通用能力。

### 5.1 输入参数

历史工作流中起始节点包含：

- `subject`：选题
- `wenan`：用户自定义文案
- `account_name`：账号水印
- `fangxie`：是否仿写
- `api_key`：外部能力调用 key
- `wenan_txt`：可选文案文件

这些字段说明模板需要有明确的输入 Schema，而不是让参数散落在节点中。

### 5.2 AI 文案与翻译

历史流程中包含大模型节点，用于：

- 默认生成心理学口播稿
- 用户输入文案后的仿写
- 分镜生成
- 英文字幕翻译
- 结构化 JSON 输出

这类能力应抽象为 `LLMPlugin`。

### 5.3 分镜与字幕结构

历史流程中出现了：

- `all_fenjing_list`
- `curr_fenjing_list`
- `cap_list`
- `time_list`
- `cap_time_list`
- `wenanTimeline`

这些是非常关键的中间结构，应沉淀为统一的内部对象，例如：

- `Storyboard`
- `Scene`
- `CaptionSegment`
- `TimelineItem`

### 5.4 配图与去重

历史流程中存在：

- 循环选择火柴人图片
- 提取配图链接
- 按文件名去重
- 按分镜分配图片轨道

这类能力应抽象为 `ImagePlugin` 和 `AssetResolver`。

### 5.5 语音合成

历史流程中使用语音插件生成配音 mp3，包含：

- 输入文本
- 音色 ID
- 语速
- 情绪参数
- 返回音频链接
- 返回音频时长

这类能力应抽象为 `TTSPlugin`，统一返回 `AudioAsset`。

### 5.6 剪映草稿插件

历史流程中使用了多类剪映草稿插件方法：

- `create_draft`
- `add_images`
- `add_captions`
- `add_audios`
- `add_keyframe`

这些能力应该成为底层 Draft Adapter，而不是散落在模板业务逻辑中。

## 6. 总体技术路线

### 6.1 目标链路

整体链路如下：

```text
用户输入参数
  ↓
Input Schema 校验
  ↓
LLM 生成 / 改写文案
  ↓
Storyboard 生成分镜
  ↓
Image Plugin 生成 / 匹配配图
  ↓
TTS Plugin 生成配音 mp3
  ↓
Caption Plugin 生成字幕
  ↓
Timeline Plugin 对齐字幕、图片、音频
  ↓
Template Builder 组装内部 VideoProject
  ↓
Draft Adapter 转为 CapCut Mate Payload
  ↓
CapCut Mate / 剪映草稿服务生成草稿 DSL/JSON
  ↓
Draft Package 打包素材和 manifest
  ↓
Local Draft Helper 本地转换为剪映草稿目录
  ↓
剪映打开预览、二次剪辑、导出
```

### 6.2 核心原则

- 模板逻辑高代码化
- 通用能力插件化
- 内部结构 DSL 化
- 底层剪映适配隔离化
- 模板 Spec 文档化
- 输入输出可测试化
- 生成过程可追踪化
- 中间产物可回放化

## 7. 系统分层设计

建议系统分为 5 层。

### 7.1 Template Spec 层

负责描述模板：

- 模板名称
- 模板适用场景
- 视频结构
- 参数要求
- 素材规则
- 节奏规则
- 字幕规则
- 输出规则
- 验收标准
- 维护说明

这一层面向产品、运营、研发和 AI Coding。

### 7.2 Template Builder 层

负责高代码编排。

它根据输入参数和模板规则，调用通用插件能力，最终生成内部 `VideoProject`。

模板 Builder 不应该直接拼接剪映草稿 JSON，也不应该直接依赖某个具体平台插件。

### 7.3 Capability Plugin 层

负责所有通用能力，包括：

- LLM 文案插件
- 分镜插件
- 配图插件
- TTS 插件
- 字幕插件
- 时间线插件
- 动画插件
- 文件处理插件
- 图片处理插件

插件返回统一资产对象，例如 `TextAsset`、`ImageAsset`、`AudioAsset`、`CaptionSegment`。

### 7.4 Video Workflow DSL 层

内部标准视频结构，负责表达：

- 项目信息
- 画布尺寸
- 场景
- 轨道
- 图层
- 素材
- 字幕
- 音频
- 动画
- 时间线
- 样式

这是系统最核心的中间结构。

### 7.5 Draft Adapter 层

负责把内部 `VideoProject` 转换为剪映草稿 DSL/JSON 或 CapCut Mate Payload。

这一层负责适配：

- `create_draft`
- `add_images`
- `add_captions`
- `add_audios`
- `add_keyframe`
- 草稿路径
- 素材路径
- 剪映版本兼容

## 10. 剪映草稿技术路线

### 10.1 引入 capcut-mate

当前计划引入开源项目 `capcut-mate` 作为底层剪映草稿生成能力。

技术路线：

```text
内部 VideoProject
  ↓
CapCutMateAdapter
  ↓
capcut-mate payload
  ↓
剪映草稿 JSON / DSL
  ↓
本地工具转换为剪映本地草稿文件
```

### 10.2 为什么不直接生成最终视频

不直接渲染视频的原因：

- 在线渲染成本高
- 视频导出耗时长
- 用户仍希望在剪映中二次修改
- 剪映已经提供成熟的预览、剪辑、导出能力
- 当前目标是生成“草稿”，不是做完整云剪辑平台

### 10.3 草稿包设计

建议生成统一草稿包：

```text
draft_package.zip
├── manifest.json
├── project.workflow.json
├── capcut_mate_payload.json
├── assets/
│   ├── image_001.png
│   ├── image_002.png
│   ├── voice_001.mp3
│   ├── bgm_001.mp3
│   └── logo.png
└── README.md
```

### 10.4 本地转换工具

由于剪映草稿依赖本地路径，建议提供本地工具：

```text
Local Draft Helper
  ↓
下载草稿包
  ↓
解压素材
  ↓
重写本地素材路径
  ↓
生成剪映草稿目录
  ↓
提示用户在剪映中打开
```

第一阶段优先支持 Windows 剪映专业版。

## 14. 高代码 Builder 示例

```python
class PsychologyStickmanTemplate(VideoTemplate):
    code = "psychology_stickman_001"

    async def build(self, ctx: BuildContext) -> VideoProject:
        input_data = PsychologyInput.model_validate(ctx.input)

        # 1. 文案生成或读取
        script = await self.resolve_script(ctx, input_data)

        # 2. 分镜生成
        storyboard = await ctx.plugins.storyboard.from_script(
            script=script,
            rule={
                "min_scene_count": 8,
                "max_scene_count": 16,
                "required_fields": ["cap", "start", "end", "desc_keywords", "changjing", "qingxufenwei"]
            }
        )

        # 3. 配图
        images = await ctx.plugins.image.resolve_for_scenes(
            scenes=storyboard.scenes,
            style="stickman",
            dedupe=True
        )

        # 4. 配音
        voice = await ctx.plugins.tts.synthesize(
            text=script.full_text,
            voice_id=input_data.voice_id or "default_young_male",
            speed=1.1
        )

        # 5. 字幕
        zh_captions = ctx.plugins.caption.build_from_storyboard(
            storyboard=storyboard,
            style_id="zh_top_black"
        )

        en_texts = await ctx.plugins.llm.translate(
            text=script.full_text,
            target_lang="en"
        )

        en_captions = ctx.plugins.caption.build_bilingual(
            source=zh_captions,
            translated_texts=en_texts,
            style_id="en_bottom_black"
        )

        # 6. 组装 VideoProject
        project = self.compose_project(
            input_data=input_data,
            storyboard=storyboard,
            images=images,
            voice=voice,
            zh_captions=zh_captions,
            en_captions=en_captions
        )

        # 7. 校验
        ctx.plugins.validator.validate_project(project)

        return project
```

## 17. 错误处理

### 17.1 常见错误

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 输入错误 | 字幕过长、缺少图片 | 提前校验 |
| LLM 错误 | 输出 JSON 不合法 | 自动修复 / 重试 |
| 图片错误 | 图片生成失败 | fallback 到图库 |
| TTS 错误 | 语音合成失败 | 重试 / 更换音色 |
| 时间线错误 | 音频和字幕不对齐 | 重新分配时间线 |
| 草稿错误 | 剪映打不开 | 保存 payload 与素材包，方便回放 |
| 路径错误 | 本地素材路径失效 | Local Helper 重写路径 |

### 17.2 降级策略

- LLM 失败：使用用户原文或模板默认文案
- 配图失败：使用默认背景 / 默认图标
- TTS 失败：生成无配音草稿
- 英文字幕失败：只生成中文字幕
- 关键帧失败：取消动画但保留主体内容
- 草稿生成失败：保留内部 VideoProject 和错误日志

## 22. AI Coding 协作规范

为了让 AI Coding 更好参与，应保证每个模板有清晰上下文。

- `spec.md`
- `input_schema.json`
- `builder.py`
- `sample_input.json`
- `handoff.md`
- 失败日志或期望变更

### 22.2 AI 可执行任务

AI 适合做：

- 根据 Spec 生成 Builder
- 根据 Schema 生成前端表单
- 根据 sample input 生成测试用例
- 新增模板变体
- 调整字幕样式
- 调整节奏分配
- 迁移低代码节点逻辑
- 生成 Draft Adapter 映射
- 检查潜在越界风险

AI 不应在没有验收的情况下直接：

- 修改底层 Draft Adapter
- 修改剪映草稿兼容逻辑
- 增加未经验证的剪映特效
- 随意改变模板 scene 数量和时间结构
- 移除核心输入字段

## 23. 关键风险

### 23.1 剪映草稿格式兼容风险

剪映草稿格式不是稳定公开 API，不同版本可能存在差异。

应对方式：

- 固定剪映版本白名单
- 建立草稿生成回归测试
- 保留失败草稿包
- 为每个模板建立剪映打开测试

### 23.2 本地路径风险

草稿文件依赖本地素材路径。

应对方式：

- 云端只生成草稿包
- 本地工具负责解压和路径重写
- 所有素材统一进入 assets 目录
- manifest 记录素材映射

### 23.3 字体 / 特效风险

剪映本地不一定有某些字体、花字、气泡、特效。

应对方式：

- 第一阶段使用基础字体和稳定样式
- 建立样式白名单
- 不依赖剪映付费素材
- 模板上线前做兼容测试

### 23.4 插件供应商风险

LLM、TTS、图片生成、配图服务都可能变化。

应对方式：

- 插件接口抽象
- 供应商可替换
- 调用结果缓存
- 失败降级
- 调用日志审计

## 25. 总结

本方案的核心不是再搭一个低代码工作流，也不是做一个完整剪辑器，而是：

> 把爆款视频生产流程沉淀成可维护、可测试、可复用、可 AI Coding 的高代码模板工程体系。

最终形成的核心资产包括：

- 模板 Spec
- 输入 Schema
- 高代码 Builder
- 通用能力插件
- 内部 Video DSL
- 剪映草稿 Adapter
- 本地草稿转换工具
- 模板测试与验收规范

这套体系能够承接过去在扣子 / n8n 中沉淀的复杂经验，同时避免低代码节点图在长期维护、版本升级、工程测试和 AI Coding 上的局限。

一句话概括：

> 低代码用于验证流程，高代码用于沉淀产品。  
> 我们要做的是“爆款视频模板到剪映草稿”的自动化生产线。
