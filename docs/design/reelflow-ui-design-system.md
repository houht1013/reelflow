# Reelflow UI 设计规范

> 状态：设计基准草案  
> 适用阶段：MVP UI 重构及后续迭代  
> 适用对象：产品、设计、前端、运营  
> 主应用：`apps/tanstack-app`

## 1. 目标

本规范用于指导 Reelflow 前端 UI 重构，目标是建立独立于 TinyShip 模板的产品视觉和交互体系。

Reelflow 是面向短视频创作的 SaaS 工作台，核心用户包括非技术个人创作者、小团队和运营人员。界面必须降低理解成本，让用户快速知道能做什么、下一步点哪里、任务是否在正常执行、产物是否可以交付。

本规范重点解决：

- 摆脱脚手架和模板站视觉。
- 避免把需求、实现、技术架构说明直接展示给普通用户。
- 建立清晰的信息架构和导航模型。
- 统一落地页、工作台、任务、模板、资产和账户页面的视觉语言。
- MVP 优先交付 PC 工作台效率，移动端作为后续迭代规划保留设计方向。
- 为设计和前端提供可执行的页面与组件标准。

## 2. 产品气质

Reelflow 的应用内气质定义为：

**克制的创作工作台 + 局部创作感表达。**

关键词：

- 清晰
- 可信
- 有秩序
- 有创作感
- 不技术化
- 不模板站味

工作台主体应接近 Linear、Stripe、shadcn dashboard 这类清晰、安静、可重复操作的工具体验；模板封面、产物预览、生成进度、官方案例等区域可以带有更强的短视频创作感。

不采用：

- 过重暗色科技风。
- 大面积 AI 紫蓝渐变。
- 满屏霓虹、玻璃拟态、光效装饰。
- 营销大字报式工作台。
- 把产品需求、实现方案、Provider、Worker、MVP 等内部语言展示给用户。

## 3. 视觉分层

Reelflow 分为两套界面场景，但共享品牌 token。

### 3.1 落地页

落地页负责获客、转化和建立产品期待。

可使用更强视觉表达：

- 首屏品牌表达。
- 模板效果预览。
- 官方精选案例。
- 定价和订阅权益。
- FAQ。
- 用户价值和使用流程。
- 适度动效。

可参考 Magic UI Pro 的落地页结构和动效表达，但不能照搬为工作台风格。

### 3.2 登录后工作台

工作台负责让用户会用、敢用、复用。

应保持：

- 信息结构清楚。
- 状态反馈明确。
- 操作路径短。
- 表单和列表易扫描。
- 卡片和表格密度适中。
- 动效服务于状态变化，不做装饰性炫技。

可参考 shadcn/studio 的 dashboard、sidebar、data display、form、settings 类区块。

## 4. 色彩方向

默认色彩方向：

**石墨黑 + 珊瑚红点缀 + 琥珀状态色 + 冷白工作台。**

### 4.1 浅色模式

浅色是默认体验，优先打磨。

建议 token 语义：

| Token | 用途 | 建议方向 |
| --- | --- | --- |
| `background` | 页面背景 | 冷白、浅灰白 |
| `surface` | 面板、卡片 | 白色 |
| `foreground` | 主要文字 | 石墨黑 |
| `muted` | 次要背景 | 浅灰 |
| `muted-foreground` | 次要文字 | 中性灰，必须保证可读 |
| `border` | 分割线、卡片边界 | 低对比浅灰 |
| `primary` | 品牌强调、主 CTA | 珊瑚红 / 番茄红 |
| `primary-foreground` | 主 CTA 文字 | 白色 |
| `warning` | 积分、欠费、提醒 | 琥珀色 |
| `success` | 完成、可用 | 低饱和绿色 |
| `info` | 运行中、处理中 | 蓝色或靛蓝 |
| `destructive` | 失败、危险操作 | 错误红，与品牌红区分 |

### 4.2 深色模式

支持夜间模式，但不是另一套强视觉。

深色模式要求：

- 可用、可读、状态清晰。
- 不做黑金风、霓虹风、AI 炫光风。
- 主色降低刺眼程度。
- 琥珀和红色在深色下提高对比但降低饱和。
- 边框和分割线在深色下必须可见，但不能发亮。

深色模式的主要使用场景：

- 夜间查看任务状态。
- 预览图片、视频、音频产物。
- 长时间操作工作台。

## 5. 字体与排版

MVP 阶段优先使用系统字体栈，保证中英文和数字显示稳定。

排版原则：

- 正文最小 14px，移动端表单和正文优先 16px。
- 页面标题克制，不使用营销页级别大字号进入工作台。
- 数据、积分、任务耗时、百分比使用等宽或 tabular number。
- 中文文案行高保持 1.5-1.7。
- 卡片标题 16-18px；表格和列表标题 14-16px。
- 工作台中避免超长段落，长说明进入帮助、抽屉或折叠区。

## 6. 圆角、阴影和密度

默认圆角：

- 常规按钮、输入框、选择器：6-8px。
- 卡片：8px。
- 大预览面板：8-12px。
- 模态框和抽屉：8-12px。

避免：

- 大量 20px 以上圆角。
- 卡片套卡片。
- 装饰性重阴影。
- 模糊玻璃背景作为工作台常规底色。

密度原则：

- 工作台应适合扫描和重复操作。
- MVP 不以完整移动端体验作为验收项，但基础响应式不能出现内容重叠和不可操作。
- PC 端允许更高信息密度，但不得把表单、日志、结果、说明堆在同一区块。

### 6.1 组件实现原则

Reelflow 不从 0 到 1 重造基础组件和交互。前端实现优先采用当前工程已有的 shadcn/Radix 组件体系，包括 Button、Badge、Alert、Dialog、Progress、Tooltip、Form、Select 等基础能力。

引入开源组件库的原则：

- 基础交互优先复用 shadcn/Radix，不再额外引入一套通用 UI 框架。
- 当需要复杂表格、图表、画布、时间线、富文本等专项能力时，可以引入成熟开源库，但必须先确认与当前 React、TanStack、Tailwind、i18n 和主题体系兼容。
- 不在页面里散落一次性手写组件样式。高频面板、列表项、空状态、状态胶囊、媒体预览应沉淀为可复用组件或统一 class。
- 不照搬模板示例文案、需求说明、技术实现说明作为用户界面文案。

当前工作台 surface 规则：

- 页面背景使用低对比冷白层次，不使用纯白大平面。
- 主要面板使用 `reelflow-panel`，通过低对比 hairline、轻阴影和浅背景区分层级。
- 面板内的指标块、资产卡、阶段卡使用 `reelflow-muted-tile` 或 `reelflow-soft-tile`。
- 避免大面积 `border bg-card shadow-sm` 的脚手架默认卡片组合。
- 分隔线只用于列表和表格的扫描辅助，颜色必须低对比。

## 7. 信息架构

### 7.1 桌面端

桌面端采用：

**左侧 Sidebar + 顶部轻 Header。**

Sidebar：

- 承担主要导航。
- 支持分组。
- 当前页高亮。
- 管理端仅管理员可见。

顶部 Header：

- 当前工作区。
- 积分余额。
- 进行中任务提示。
- 通知。
- 用户菜单。
- 主题切换。

### 7.2 移动端，后续迭代

移动端完整体验不进入 MVP 当前实现范围。已讨论出的方向保留为后续迭代依据，不在第一版强行压缩桌面工作台。

后续移动端可采用：

**底部 Tab + 页面顶部标题/返回。**

底部 Tab：

- 首页
- 创作
- 任务
- 资产
- 我的

移动端不是桌面端压缩版。复杂表格、管理端、技术日志优先 PC；移动端后续只保留核心使用路径。

MVP 当前要求：

- PC 端必须完整可用。
- 移动端只保留基础响应式兜底，不作为核心验收。
- 不实现移动底部 Tab、移动专属工作台和移动管理端。
- 不删除移动端设计讨论结果，后续迭代从本节继续展开。

## 8. 一级导航

最终一级导航：

- 首页
- 创作，可展开二级菜单
- 任务
- 模板
- 资产
- 账户
- 管理，仅管理员可见

桌面端 Sidebar 的主业务导航顺序：

1. 首页，独立一级入口。
2. 创作，一级可展开菜单。
3. 任务，一级入口。
4. 模板，一级入口。
5. 资产，一级入口。

创作的二级菜单：

- 短视频草稿。
- 图像生成。
- 视频生成，MVP 占位，显示待开放。
- 语音生成，MVP 占位，显示待开放。

### 8.1 首页

首页是已登录用户的工作台门户，不是公开落地页。

首页用途：

- 功能快捷入口。
- 营销信息和活动推荐。
- 相关推广。
- 官方精选案例画廊。
- 用户状态摘要。

首页不做强广告页，应采用工作台内的运营推荐卡片。

首页模块优先级：

1. 快捷入口。
2. 进行中任务和积分状态。
3. 官方精选案例。
4. 新模板和推荐赛道。
5. 邀请、订阅、定制服务。

官方精选案例只展示官方精选，不混入用户个人历史产物。用户个人产物后续可单独增加“你的最近作品”模块。

### 8.2 创作

创作作为 Sidebar 一级可展开菜单承载创作能力，不再同时作为工作台分组和独立分组重复出现。

创作菜单包含：

- 短视频草稿。
- 图像生成。
- 视频生成，MVP 占位，显示即将开放。
- 语音生成，MVP 占位，显示即将开放。

MVP 实现：

- 短视频草稿。
- 图像生成。

不在资产库中主导 AI 生图、AI 视频、AI 语音入口。资产库主要负责纳管生成结果和用户上传素材。

### 8.3 任务

任务用于追踪和管理用户提交的短视频草稿生成任务。

包含：

- 任务列表。
- 任务详情。
- 处理记录入口，单独设计，不揉进任务详情主页面。

### 8.4 模板

模板库独立保留，承载官方模板和私有模板。

包含三层：

1. 模板列表。
2. 模板详情。
3. 使用模板创建。

### 8.5 资产

资产库主要用于纳管：

- 任务执行产物。
- 用户上传素材。
- AI 生成资产。

默认素材画廊，提供列表视图切换。

### 8.6 账户

账户相关能力可在 Sidebar 分组展示，移动端后续集中到“我的”。

包含：

- 积分。
- 订阅。
- 邀请。
- 通知。
- 设置。

## 9. 关键流程

### 9.1 短视频草稿创建流程

流程：

1. 首页或创作中心进入短视频草稿。
2. 选择模板。
3. 查看模板详情。
4. 点击使用模板。
5. 填写参数。
6. 预估积分和预检。
7. 开始生成。
8. 跳转任务详情。

原则：

- 先选模板，再填参数。
- 不把模板和参数并列堆在同一复杂页面。
- 不提供工作流节点编辑器。
- 不做分步生成。
- 支持示例填充。

### 9.2 图像生成流程

图像生成是独立创作能力。

MVP 页面结构：

- 左侧或上方：描述输入、画面比例、风格、数量、积分预估。
- 右侧或下方：结果预览画廊。
- 历史生成结果。
- 主 CTA：生成图片。

原则：

- 默认参数简单。
- 高级参数折叠。
- 不向普通用户展示 provider/model，除非后续设计高级模式。
- 生成结果自动进入资产库。

### 9.3 视频生成和语音生成

MVP 先作为占位能力。

占位页要求：

- 清楚说明该能力即将开放。
- 可展示能力方向和示例。
- 可提供预约、订阅提醒或联系定制服务入口。
- 不展示未完成的技术字段。

## 10. 页面规范

### 10.1 首页

首页第一屏建议包含：

- 欢迎标题和简短引导。
- 快捷入口卡：创建草稿、图像生成、查看任务、购买积分。
- 状态摘要：积分余额、冻结积分、进行中任务、待处理问题。
- 官方精选案例横向画廊。
- 推荐卡：新模板、推荐赛道、邀请送积分、定制私有模板服务。

设计要求：

- 不做大面积营销 Banner。
- 推荐信息以工作台卡片呈现。
- 入口卡片清晰，不要在卡片内写需求或实现说明。

### 10.2 创作中心

创作中心使用 4 个能力入口卡：

- 短视频草稿。
- 图像生成。
- 视频生成。
- 语音生成。

卡片内容：

- 能力名称。
- 一句话用户价值。
- 简短场景。
- 状态标签：可用 / 即将开放。
- CTA。

禁止展示：

- Provider。
- 工作流编排。
- MVP。
- 技术栈。
- 内部实现方式。

### 10.3 模板列表

模板列表采用视频封面卡，而不是工具功能卡。

列表能力：

- 官方模板 / 私有模板筛选。
- 分类筛选：知识分享、观点表达、情绪价值等。
- 标签：推荐、低素材要求、适合起号、可定制。
- 搜索。

模板卡包含：

- 封面图或短视频风格预览。
- 模板名称。
- 适用场景。
- 输出类型：剪映草稿、可选 MP4。
- 预计积分。
- 预计时长。
- 官方推荐 / 私有等标签。
- CTA：使用模板。

模板卡禁止展示：

- 工作流步骤。
- 模型或 provider。
- 内部参数 schema。
- 实现方式。

### 10.4 模板详情

模板详情包含：

- 模板效果预览。
- 适合场景。
- 输入参数说明。
- 输出内容。
- 预计积分。
- 使用案例。
- 定制服务入口。
- CTA：使用模板。

模板详情应帮助用户判断“这个模板适不适合我”，不是解释“这个模板怎么实现”。

### 10.5 创建参数页

页面结构：

- 顶部：当前模板摘要。
- 主区：参数表单。
- 侧区：输出说明、预计积分、注意事项。
- 底部固定 CTA：开始生成。

必须支持：

- 示例填充。
- 清空。
- 必填标记。
- 字段级提示。
- 积分预估。
- 预检错误展示。
- 提交中状态。

不做：

- 半成品预览。
- 节点预览。
- 高级技术参数。
- 分步执行。

### 10.6 任务列表

任务列表用于让用户快速识别任务状态。

列表字段建议：

- 任务名称 / 模板。
- 状态。
- 进度。
- 创建时间。
- 完成时间。
- 预计积分。
- 实际积分。
- 产物状态。
- 质量状态。
- 操作。

状态必须用文字、图标和颜色共同表达，不能只依赖颜色。

### 10.7 任务详情

任务详情主页面不揉进技术日志。

页面结构：

- 状态概览：任务状态、进度、积分、时间。
- 阶段时间线：预检、脚本、分镜、图片、配音、字幕、草稿包、MP4、通知。
- 产物预览：草稿包、图片、音频、字幕、MP4。
- 质量问题：缺图、音频缺失、内容被拦截、需要人工补齐。
- 补齐操作：去补图、去补语音、重新生成、重试失败点。
- 下载操作：草稿包 / MP4。

处理记录：

- 提供轻入口。
- 可用右侧抽屉或单独页面。
- 普通用户看到用户可读事件。
- 管理员可看到技术日志、错误码、服务返回、用量明细。

### 10.8 资产库

默认视图：

- 素材画廊。

辅助视图：

- 列表视图。

筛选：

- 全部。
- 任务产物。
- 用户上传素材。
- AI 生成资产。
- 图片 / 音频 / 视频 / 草稿包。
- 时间。
- 来源任务。

卡片内容：

- 缩略图或类型图标。
- 类型。
- 来源。
- 任务或模板。
- 创建时间。
- 状态。

详情：

- 点击打开侧边预览面板。
- 支持图片预览、音频试听、视频预览、草稿包说明。

资产库不是 AI 工具入口主场，不应把图像生成、视频生成、语音生成作为资产库主功能。

### 10.9 积分页面

积分页面重点让用户理解：

- 当前可用积分。
- 冻结积分。
- 欠费积分。
- 购买入口。
- 流水记录。
- 任务扣减和结算。

设计要求：

- 金额和积分使用清晰数字层级。
- 欠费状态必须明确提示恢复路径。
- 购买按钮突出但不过度营销。
- 流水表支持筛选和空态。

### 10.10 订阅页面

订阅页面用于商业闭环。

内容：

- 当前套餐。
- 套餐权益。
- 周期订阅。
- 可并行任务数。
- 每月积分额度。
- 升级/续费。

后续如并行资源数量进入套餐，应在此明确展示。

### 10.11 邀请页面

MVP 不做分销结算，只做邀请赠送积分。

页面内容：

- 邀请链接。
- 邀请码。
- 双方奖励。
- 成功邀请数。
- 奖励到账记录。

不要展示“分销结算中心”等尚未实现承诺。

### 10.12 通知页面

通知分类：

- 任务完成。
- 任务失败。
- 积分到账。
- 积分欠费。
- 资产可用。
- 邀请奖励。

通知优先站内通知和邮件。

## 11. 组件规范

### 11.1 基础组件

继续使用当前技术栈：

- Tailwind CSS v4。
- shadcn/ui 风格本地组件。
- Radix UI primitives。
- lucide-react icons。
- class-variance-authority。
- clsx + tailwind-merge。

优先复用：

- `libs/react-shared/ui/*`
- `libs/ui/styles/*`

可参考 shadcn/studio 区块，但应复制后改造为 Reelflow 风格，不直接保留模板文案和模板结构。

### 11.2 图标

规则：

- 使用 lucide-react。
- 不使用 emoji 作为结构性图标。
- 同一层级保持线性图标风格一致。
- 图标按钮必须有可访问名称。

### 11.3 卡片

卡片类型：

- 模板封面卡。
- 快捷入口卡。
- 状态摘要卡。
- 资产卡。
- 推荐卡。
- 定价/订阅卡。

规则：

- 不做卡片套卡片。
- 卡片圆角默认 8px。
- 卡片内信息层级清晰。
- 卡片可点击时必须有 hover、focus、pressed 状态。

### 11.4 表单

规则：

- 必须有可见 label。
- 不用 placeholder 代替 label。
- 错误提示展示在字段附近。
- 复杂字段给短 helper text。
- 提交中禁用按钮并显示状态。
- 表单底部固定主 CTA；移动端固定 CTA 后续专项设计。

### 11.5 状态标签

任务状态：

- 排队中。
- 运行中。
- 已完成。
- 失败。
- 已取消。
- 待补齐。
- 可下载。
- 欠费锁定。

阶段状态：

- 未开始。
- 进行中。
- 已完成。
- 已跳过。
- 需要补齐。
- 失败。

状态展示必须包含：

- 文案。
- 图标或形状。
- 颜色。

### 11.6 空态

每个主要页面必须设计空态。

空态包含：

- 当前没有什么。
- 用户下一步可以做什么。
- 一个明确 CTA。

示例：

- 无任务：去创建第一个草稿。
- 无资产：运行任务或上传素材后会显示在这里。
- 无模板：联系管理员发布模板。
- 无邀请：复制链接邀请好友。

### 11.7 加载和错误

加载：

- 列表和卡片优先 skeleton。
- 按钮异步操作显示 loading。
- 长任务显示进度，不使用无限 spinner 作为主要反馈。

错误：

- 明确原因。
- 明确恢复路径。
- 可重试时提供重试按钮。

## 12. 响应式规则

MVP 响应式策略：PC 优先。1024px 及以上作为当前主要验收范围；375px、768px 只做基础可读和不阻塞访问，不要求完整移动工作台体验。

### 12.1 断点

建议断点：

- 375px：小屏手机。
- 768px：平板 / 大手机横屏。
- 1024px：桌面。
- 1440px：宽屏。

### 12.2 移动端页面规则，后续迭代

以下内容为后续移动端专项设计记录，当前 MVP 不纳入完整实现验收。

首页：

- 卡片纵向排列。
- 快捷入口优先。

创作中心：

- 大入口卡片。
- 未开放能力灰态。

模板库：

- 手机单列封面卡。
- 大屏手机可双列。

创建参数：

- 单列表单。
- 底部固定生成按钮。

任务详情：

- 状态概览优先。
- 阶段时间线纵向。
- 产物预览往下。

资产库：

- 默认画廊。
- 筛选收进抽屉。

我的：

- 聚合积分、订阅、邀请、通知、设置。

管理端：

- 移动端可提示建议在电脑端使用。

## 13. 动效规则

动效服务于状态和空间关系。

允许：

- 页面切换轻微 fade/slide。
- 卡片 hover/pressed。
- 下拉、抽屉、弹窗进入退出。
- 任务进度更新。
- Skeleton shimmer。
- 生成结果出现时轻微入场。

避免：

- 装饰性背景动效。
- 长时间循环光效。
- 大面积渐变流动。
- 影响阅读的动效。
- 超过 500ms 的常规 UI 动效。

应支持 `prefers-reduced-motion`。

## 14. 文案规范

用户界面文案必须使用用户语言。

允许：

- 描述用户价值。
- 描述当前状态。
- 描述下一步动作。
- 描述可恢复路径。

禁止在普通用户界面展示：

- MVP。
- Demo。
- 实现。
- 可扩展设计。
- Provider。
- Worker。
- Runtime。
- Route protection。
- Payload。
- Schema。
- 内部验收标准。
- 需求描述。
- 技术架构说明。

管理端技术页面可以在必要时使用运营术语，但仍应避免把工程实现解释当作页面介绍。

### 14.1 前端文案边界，硬性规范

以下规则是发布门禁，不是文案建议：

1. 需求文档、产品讨论、技术方案、架构说明、代码注释、TODO 和验收标准不能直接或改写后作为普通用户界面说明。
2. 普通用户页面不得直接渲染后端返回的 `message`、`error`、事件日志、错误码、状态枚举或资产备注。必须先映射为可理解的状态、原因和恢复动作。
3. 普通用户页面不得展示 Provider、模型标识、Worker、Runtime、存储服务、存储键、Payload、Schema、路由保护、队列锁、成本字段等内部实现信息。
4. 未完成功能只能说明用户可理解的可用状态，例如“即将开放”或“暂不可用”，不能出现“MVP 占位”“尚未实现”“后续接入”“可扩展设计”等研发进度说明。
5. 页面是否出现在导航中不影响本规则。只要路由可被直接访问，就必须满足消费者文案要求，或限制为管理员访问。
6. SEO 标题、描述、Toast、Dialog、Tooltip、空态、错误态、无障碍标签和图片替代文本同样属于用户界面文案。

文案必须回答以下至少一项：

- 用户现在能获得什么。
- 当前处于什么状态。
- 为什么暂时不能继续。
- 下一步可以做什么。

不能回答这些问题、只是在解释系统如何实现的文字，应删除或移入管理员页面、开发文档和日志系统。

### 14.2 文案受众分级

| 受众 | 允许内容 | 禁止内容 |
| --- | --- | --- |
| 公开访客 | 产品价值、案例、价格、使用流程、服务说明 | 工作台功能堆叠、内部状态、技术架构、研发计划 |
| 普通用户 | 任务状态、进度、产物、用量、恢复动作 | 原始日志、Provider/模型、存储信息、原始错误、成本明细 |
| 管理员 | 运维状态、服务健康、成本、错误详情、执行锁 | 把需求说明或代码实现说明当作页面介绍 |

完整检查方法和现有问题记录见
`docs/design/frontend-copy-boundary.md`。

普通用户：

- 首页。
- 创作。
- 任务。
- 模板。
- 资产。
- 积分。
- 订阅。
- 邀请。

## 18. 交付检查清单

设计交付前检查：

- 是否符合“克制的创作工作台 + 局部创作感表达”。
- 是否默认浅色，并完整考虑深色适配。
- MVP 是否采用左侧 Sidebar；移动底部 Tab 仅作为后续迭代规划。
- 首页是否是工作台门户，而不是广告页。
- 创作中心是否包含四个能力入口。
- 模板是否以视频封面卡呈现。
- 任务详情是否聚焦状态、阶段、产物和补齐操作。
- 资产库是否默认画廊，并支持列表切换。
- 是否没有把需求或实现说明展示给普通用户。
- 是否为加载、错误、空态、无权限状态设计了界面。
- 移动端后续是否是可用工作台，而不是桌面端压缩版。

前端实现前检查：

- 页面文案必须进入 i18n。
- 不硬编码用户可见文本。
- 不使用需求文档、技术方案、代码注释或接口字段作为页面文案来源。
- 不直接展示后端原始错误、事件消息、状态枚举、Provider、模型或存储标识。
- 所有可直接访问的普通用户路由均完成文案审查，不能以“导航已隐藏”为理由跳过。
- 使用 shadcn/Radix/Tailwind 现有组件体系。
- 用 token 而不是散落硬编码颜色。
- 每个可交互元素有 hover、focus、disabled 状态。
- 图标使用 lucide-react。
- 不使用 emoji 作为结构性图标。
- MVP 至少验证 1024px、1440px 桌面；375px、768px 仅做基础不重叠检查，完整移动端后续专项验收。
- 任务状态不能只靠颜色表达。
- 普通用户界面不得出现内部技术术语。
